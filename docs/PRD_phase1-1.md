## 1. 개요

Phase 0는 계산기 수준의 시뮬레이터였다. Phase 1-1은 **실제 공공 데이터와 공공 API**를 연결해, 사용자가 "이 서비스가 내 주소를 알고 있다"는 실감을 얻게 만드는 단계다.

결제 없이, Apify 없이, 공개된 외부 데이터만 사용한다.

핵심 인식 전환 목표:

> "수익 계산기" → "내 입지를 분석해주는 도구"

### 3-1. 페이지 레이아웃 재배치 (P0)

#### 목표
현재 레이아웃은 시뮬레이터가 전면에 있고 이메일 수집은 하단에 배치되어 있다. Phase 1-1에서는 이메일 등록 유도 흐름을 강화하도록 재구성한다.

#### 변경 전 (Phase 0)
```
Navbar
HeroSection (이메일 수집 폼)
SimulatorSection (수익 계산기)
ResultCards
MonthlyLedger
ComingSoonSection
Footer
```

#### 변경 후 (Phase 1-1)
```
Navbar
HeroSection (카피 + CTA: "내 주소로 분석하기")
BuildingCheckSection (건축물대장 조회 — 새 기능)
  └─ 조회 결과 → 외도민 등록 가능성 추정 뱃지
  └─ 결과 하단: 이메일 수집 CTA (1차)
CompetitionSection (경쟁밀도 — 새 기능)
  └─ "반경 500m 내 경쟁 숙소 N개"
  └─ 결과 하단: 이메일 수집 CTA (2차)
SimulatorSection (수익 계산기 — 기존 + 운영 유형 선택 추가)
  └─ 결과 하단: 이메일 수집 CTA (3차)
ComingSoonSection
Footer
```

#### 이메일 CTA 배치 원칙
- 각 기능 결과 노출 직후 이메일 CTA를 삽입한다.
- "더 정밀한 분석을 원하시면 → 출시 알림 신청" 문구 사용.
- 이미 등록한 사용자에게는 CTA를 "등록 완료" 상태로 전환 (세션 기준).

#### 완료 기준
- 새 레이아웃이 모바일/데스크탑에서 렌더링 오류 없이 동작
- 이메일 CTA가 각 섹션 결과 직후 노출됨

### 3-2. 건축물대장 API (세움터) 연동 (P0)

#### 목표
사용자가 주소를 입력하면 건축물대장 정보를 조회하고, 해당 건물이 외도민(외국인관광도시민박업) 등록 가능성이 있는지 1차 추정 결과를 보여준다.

#### 사용 API
- **세움터 건축물대장 API** (공공데이터포털 제공)
  - 엔드포인트: `https://apis.data.go.kr/1613000/ArchPmsService_v2/getApBasisOulnInfo`
  - 인증: 공공데이터포털 발급 API 키 (서버 환경변수로 관리)
  - 주요 반환 필드: 건물용도, 건물구조, 층수, 사용승인일, 대지면적, 연면적

#### 외도민 등록 가능성 추정 로직

| 조건 | 판단 |
|------|------|
| 건물 용도: "단독주택" 또는 "다가구주택" | 가능성 있음 |
| 건물 용도: "공동주택(아파트/연립/다세대)" | 등록 어려울 수 있음 (집합건물은 호별 동의 필요) |
| 건물 용도: "근린생활시설" 등 비주거 | 등록 어려울 수 있음 |
| 건물이 조회되지 않음 | 확인 필요 |

> 주의: 용도 외에도 면적, 세대 수, 지자체 조례 등에 따라 달라질 수 있으므로 결과는 참고용이다.

#### 사용자 플로우

```
주소 입력 (도로명주소 또는 지번주소)
  └─ "건물 조회" 버튼 클릭
       └─ 서버: 주소 → 좌표 변환 (카카오 주소 API)
            └─ 서버: 세움터 API 호출 (건물 기본개요 조회)
                 └─ 결과 표시:
                      ├─ 건물 기본 정보 카드
                      │    (건물용도, 층수, 사용승인일)
                      └─ 외도민 등록 가능성 뱃지
                           ├─ "가능성 있음" (녹색)
                           ├─ "등록 어려울 수 있음" (주황)
                           └─ "확인 필요" (회색)
```

#### 주소 입력 처리
- 카카오 주소검색 API(Daum Postcode)로 정형 주소 수집
- 도로명주소/지번주소 모두 세움터 API에 전달 가능

#### API 호출 위치
- **Route Handler** (`/api/building`) 에서만 호출
- API 키는 서버 환경변수 (`SEUMTER_API_KEY`) 로 관리, 절대 클라이언트 노출 금지

#### 면책 문구 (결과 카드 하단 필수 삽입)
```
이 결과는 건축물대장 기준의 참고 추정치입니다.
실제 외도민 등록 가능 여부는 관할 구청 확인이 필요합니다.
조회 기준일: [API 응답일 기준]
```

#### 완료 기준
- 주소 입력 → 세움터 API 응답 정상 수신
- 건물 기본 정보(용도, 층수, 사용승인일) 화면 출력
- 외도민 가능성 뱃지 3단계 중 하나로 표시
- 면책 문구 항상 표시
- API 키 클라이언트 미노출 확인

### 3-3. 시뮬레이터 운영 유형 선택 추가 (P0) ← NEW

#### 배경
현실에서 외도민 특례(180일) 사업자 비중이 1군(365일)보다 훨씬 높다.
기존 시뮬레이터는 365일 기준으로만 계산해 실제 수익과 최대 2배 차이가 발생할 수 있다.
사용자가 자신의 운영 유형을 선택해 현실적인 수익을 추정할 수 있도록 분기 계산을 추가한다.

#### UI 변경
시뮬레이터 상단에 운영 유형 선택 토글 추가:

```
운영 유형 선택
  ○ 1군 외도민 (외국인관광도시민박업 · 연 365일 운영 가능)
  ● 특례 사업자 (단기임대 특례 · 연 180일 상한)        ← 기본값 권장
```

> 특례 사업자를 기본값으로 설정한다. 현실 비중이 높고, 보수적 추정이 신뢰도를 높인다.

#### 계산 로직 분기

| 항목 | 1군 외도민 | 특례 사업자 |
|------|-----------|-------------|
| 연간 최대 운영일 | 365일 | 180일 |
| 월 운영일 계산 | 30일 × 예약률 | 15일 × 예약률 (180/12) |
| 연간 예상 매출 | 객단가 × 365 × 예약률 | 객단가 × 180 × 예약률 |
| ROI 기준 | 연 365일 기준 | 연 180일 기준 |

#### 결과 카드 표시
- 선택한 운영 유형을 결과 카드 상단에 명시
  - 예: "1군 외도민 기준 추정치" / "특례 사업자(180일) 기준 추정치"
- 특례 선택 시 결과 하단에 추가 안내:
  ```
  본 시뮬레이션은 외국인관광도시민박업 특례(연 180일) 기준입니다.
  1군(365일) 대비 약 50% 수준의 운영일을 적용했습니다.
  최종 확인은 관할 구청 또는 세무사에게 문의하세요.
  ```

#### 완료 기준
- 운영 유형 토글 UI 정상 동작
- 선택값에 따라 계산 결과(매출/순수익/ROI/원금회수) 즉시 재계산
- 결과 카드에 운영 유형 명시
- 특례 선택 시 면책 문구 노출

### 3-4. 경쟁밀도 수치화 (P1)

#### 목표
공공데이터 "외국인관광도시민박업" 목록을 Supabase에 적재하고, 사용자 주소 기준 반경 500m 내 경쟁 숙소 수를 보여준다.

#### 데이터 소스
- **공공데이터포털 — 외국인관광도시민박업 인허가 데이터**
  - 파일 형식: CSV (정기 업데이트)
  - 수집 범위: **서울특별시만 적재** (Phase 3 전국 확대 시 추가 적재)
  - 주요 필드: 업체명, 소재지도로명주소, 소재지지번주소, 위도, 경도, 인허가일자, 영업상태
  - 수집 주기: 최초 1회 수동 적재 (이후 Phase 2에서 자동화 검토)

> **서울만 적재하는 이유:** Phase 1-2 Apify 수집 범위가 마포/홍대/용산/이태원으로 서울 중심.
> 전국 데이터는 테이블 구조가 동일하므로 Phase 3에서 CSV 추가 적재만으로 확장 가능.

#### Supabase 테이블 설계

**테이블명: `minbak_listings`**

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| `id` | `uuid` (PK, default gen) | 고유 식별자 |
| `name` | `text` | 업체명 |
| `road_address` | `text` | 도로명주소 |
| `jibun_address` | `text` | 지번주소 |
| `lat` | `float8` | 위도 |
| `lng` | `float8` | 경도 |
| `status` | `text` | 영업상태 (영업/폐업 등) |
| `license_date` | `date` | 인허가일자 |
| `region` | `text` | 시/도 (서울특별시 등) — 향후 필터용 |
| `data_updated_at` | `date` | 원천 데이터 기준일 |
| `created_at` | `timestamptz` | 적재 일시 |

**인덱스:**
- `(lat, lng)` 복합 인덱스 — 반경 쿼리 성능용
- `region` 인덱스 — Phase 3 지역 필터용
- PostGIS 확장 사용 시 `GEOGRAPHY(POINT)` 컬럼 추가 검토

#### 반경 쿼리 방식
PostGIS 미사용 시 Haversine 공식 기반 RPC 함수로 대체:

```sql
-- Supabase RPC: get_nearby_minbak(user_lat, user_lng, radius_m)
-- 반환: 반경 내 영업 중 업체 수 (숫자만, 개별 정보 반환 금지)
```

클라이언트에서 전체 데이터를 가져와 필터링하는 방식은 **사용 금지** (성능 문제 + 개별 정보 노출 리스크).

#### 사용자 플로우

```
건축물대장 조회에서 입력한 주소 재사용 (또는 별도 입력)
  └─ "주변 경쟁 숙소 조회" 버튼 클릭
       └─ 서버: 주소 → 좌표 (카카오 API)
            └─ Supabase RPC: 반경 500m 내 minbak_listings 조회
                 └─ 결과 표시:
                      ├─ "반경 500m 내 등록 민박 N개"
                      ├─ 경쟁 강도 레이블
                      │    ├─ 0~3개: "경쟁 적음" (녹색)
                      │    ├─ 4~9개: "경쟁 보통" (주황)
                      │    └─ 10개 이상: "경쟁 치열" (빨강)
                      └─ 데이터 기준일 표시
```

#### 데이터 적재 절차 (운영자 작업)
1. 공공데이터포털에서 서울 기준 CSV 다운로드
2. 위도/경도 없는 행 제거 (또는 주소 → 좌표 변환 후 보완)
3. Supabase `minbak_listings` 테이블에 bulk insert
4. `data_updated_at` 필드에 CSV 기준일 기입
5. `region` 필드에 "서울특별시" 기입

#### 면책 문구 (결과 카드 하단 필수 삽입)
```
공공데이터 인허가 기준 집계입니다. 실제 운영 중인 숙소 수와 다를 수 있습니다.
폐업 미신고 또는 미등록 숙소(특례 사업자 포함)는 집계에서 누락될 수 있습니다.
데이터 기준일: [data_updated_at 값]
```

> **면책 문구 주의:** 특례 사업자는 공공데이터에 등록되지 않은 경우가 많다.
> "실제보다 적게 나올 수 있습니다"는 뉘앙스를 사용자에게 명확히 전달할 것.

#### 완료 기준
- `minbak_listings` 테이블에 서울 데이터 적재 완료
- 주소 입력 → 반경 500m 내 숙소 수 정상 반환
- 경쟁 강도 레이블 표시
- 면책 문구 및 데이터 기준일 항상 표시
- 개별 업체명/주소 노출 없음 (집계값만 노출)

### 3-5. 카카오톡/링크 공유 버튼 (P2)

#### 목표
시뮬레이터 결과를 지인에게 공유해 바이럴 확산을 유도한다.

#### 공유 대상 정보
- 공유 텍스트: "내 창업 예정지 분석 결과 — 방정식"
- 공유 URL: 현재 페이지 URL (쿼리 파라미터로 시뮬레이터 입력값 포함)
- 시뮬레이터 입력값은 URL 쿼리스트링으로 직렬화 (`?rent=800000&price=80000&type=special&...`)
  - `type` 파라미터: `full`(1군) / `special`(특례) — 운영 유형도 공유 URL에 포함

#### 버튼 구성
1. **카카오톡 공유** — Kakao SDK `Kakao.Share.sendDefault()` 사용
2. **링크 복사** — `navigator.clipboard.writeText(url)`

#### 공유 카드 내용 (카카오 공유)
```
타이틀: "내 창업 입지 분석 결과"
설명: "월 예상 순수익 {N}만원 | 원금회수 {M}개월 | 방정식으로 계산"
버튼: "나도 계산해보기 →"
```

#### 완료 기준
- 카카오톡 공유 버튼 클릭 → 카카오 공유 팝업 정상 동작
- 링크 복사 → 클립보드에 URL 복사 + 복사 완료 토스트 표시
- 공유 URL로 접근 시 시뮬레이터 입력값 및 운영 유형 자동 복원
