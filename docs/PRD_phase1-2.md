## 1. 개요

Phase 1-1까지는 공공 데이터와 공공 API만으로 "내 입지를 분석해주는 도구"라는 실감을 줬다.
Phase 1-2는 **돈을 받는 단계**다. 두 가지 외부 의존성이 처음으로 들어온다.

1. **AirROI API** — 에어비앤비 추정 수익/객단가/예약률 (가공된 통계 데이터)
2. **Polar** — MOR 결제 (사업자등록 없이 본인 명의 정산)

핵심 인식 전환 목표:

> "입지 분석 도구" → "들어갈지 말지 결정해주는 유료 리포트"

이 단계는 **결제·외부 API·과금**이 처음 엮이므로, 기능 구현보다 **안전장치(검증·확인창·모니터링·비용 캡)**가 더 중요하다.

## 2. 확정된 외부 의존성

| 항목 | 결정 | 근거 |
|------|------|------|
| 에어비앤비 데이터 | **AirROI API** (pay-as-you-go) | 가공 통계 제공으로 자체 점유율 계산 불필요. 한국 범위 사용·표시 허가 확인(운영자 Jason). 신생 리스크는 추상화 레이어로 방어 |
| 결제 | **Polar** (`@polar-sh/nextjs`) | MOR. 사업자등록 없이 본인 명의 정산. Next.js 14 공식 어댑터 |
| 지도 | **카카오맵** (`react-kakao-maps-sdk`) | 무료(월 300만건). 클러스터링·커스텀 오버레이 지원 |
| 공공데이터 | **`minbak_listings`** (서울, 기적재) | Phase 1-1에서 적재 완료. 외도민 인허가 |
| 건축물대장 | **세움터 API** (Phase 1-1 연동) | 기존 연동 유지 |
| 에러 모니터링 | **Sentry** (신규) | 1인 개발자가 에러를 모른 채 넘어가는 것 방지 |

> **데이터 소스 추상화 원칙:** AirROI는 신생 서비스다. `getAirbnbData()` 단일 함수로 격리해, 향후 불안정 시 이 함수 내부만 Apify 호출로 교체할 수 있게 한다. AirROI = "언제든 버릴 수 있는 소스"로 취급.

```typescript
  // 2. 캐시 미스 → AirROI API 호출
  // 3. 결과 캐시 저장 후 반환

```

#### 캐시 전략
- 동일 `(lat, lng, radius_m)` 요청은 90일간 캐시에서 반환 → AirROI 비용 0
- 캐시 만료 시 다음 요청 때만 재호출
- **AirROI 장애 시에도 캐시가 있으면 서비스는 정상 동작** (2차 방어선)

- 일일 호출 상한(예: 100건) 초과 시 → 신규 AirROI 호출 차단 + 캐시/저장 데이터만 제공 + 운영자 이메일 알림
- 비정상 호출 급증은 비용 폭탄으로 직결되므로 반드시 상한선 둠 (스레드 팁 8 반영)

#### 환경변수
- `AIRROI_API_KEY` (서버 전용, 절대 `NEXT_PUBLIC_` 금지 — 스레드 팁 2)

#### 완료 기준

### 4-2. Polar 결제 연동 (P0)

#### 목표
9,900원 단건 결제를 받고, **결제 완료 웹훅 수신 후에만** 리포트 접근 토큰을 발급한다.

#### 결제 흐름
```
[리포트 잠금 화면]
  └─ "9,900원 결제하고 정밀 리포트 보기" 버튼
       └─ 결제 직전 확인창 (스레드 팁 7)
            "9,900원이 결제됩니다. 결제 후 [주소] 분석 리포트가 열립니다."
       └─ POST /api/checkout → Polar Checkout 세션 생성 → 리다이렉트
            └─ Polar 결제 완료
                 └─ POST /api/webhooks/polar (웹훅 수신)
                      ├─ 서명 검증 (Polar webhook secret)
                      ├─ report_purchases INSERT (polar_order_id, 주소, 좌표, report_token)
                      └─ 사용자에게 리포트 URL 제공 (성공 페이지 + 이메일)
```

#### 핵심 원칙
- **토큰 발급 기준은 결제 성공 페이지가 아니라 웹훅이다.** 네트워크 이슈로 리다이렉트가 실패해도 웹훅으로 접근권이 정상 발급됨
- 웹훅 서명 검증 필수 (위조 결제 차단)
- `polar_order_id`에 UNIQUE 제약 → 웹훅 중복 수신 시 중복 발급 방지

#### 환경변수
- `POLAR_ACCESS_TOKEN` (서버 전용)
- `POLAR_WEBHOOK_SECRET` (서버 전용)
- `POLAR_PRODUCT_ID` (9,900원 리포트 상품 ID)

#### 가입 시 직접 확인 사항 (코드 외)
- Polar payout 설정에서 **South Korea + 본인 계좌 등록 가능 여부** 확인. 불가 시 Creem으로 차선 이동

#### 완료 기준
- [ ] 9,900원 결제가 실제로 작동
- [ ] 결제 직전 확인창 노출
- [ ] 웹훅 서명 검증 통과
- [ ] 웹훅 수신 → report_token 발급 → URL 제공
- [ ] 동일 주문 웹훅 중복 수신 시 중복 발급 안 됨

### 4-3. 결제 후 통합 리포트 (P0)

#### 목표
결제한 사용자에게 토큰 기반 URL로 통합 리포트를 제공한다.

#### 권한 검증 (스레드 팁 3·4 반영)
> URL을 숨기거나 로그인만으로는 보안이 아니다. **토큰 권한 체크가 핵심.**

```
GET /report/[token]
  └─ report_purchases WHERE report_token = token
       ├─ 없음 → 404 (리포트 존재 자체를 노출하지 않음)
       └─ 있음 → accessed_at 기록 → 리포트 렌더링
```

#### 리포트 구성
```
① 경쟁밀도
   - 반경 500m 내 외도민 N개 (minbak_listings)
   - 경쟁 강도 레이블 (적음/보통/치열)
   - 이 지역 평균 예약률 N% (AirROI) ← 공급 대비 수요 해석

② 건축물대장 (세움터 API)
   - 건물용도/층수/사용승인일
   - 외도민 등록 가능성 뱃지

③ AirROI 수익 데이터
   - 동네 평균 ADR / 점유율 / 월 예상 수익
   - 월별 시즌성 차트
   - 수익 구간 분포: "이 동네 상위 25% N만원 / 중간 N만원 / 하위 25% N만원"
     (개별 숙소 특정 표시 금지 — 변호사 상담 결과, 영업비밀 가능성)

④ 수익 시뮬레이터
   - 사용자 입력(월세) 대입 → 예상 순수익/ROI/원금회수기간
   - 1군(365일)/특례(180일) 토글 (Phase 1-1 로직 재사용)

⑤ 창업 가계부
   - 1~12개월차 예상 손익 (매출-월세-청소비-전기세-세금)

⑥ 면책문구 (4-7 참조)
⑦ 데이터 기준일 (AirROI 캐시 fetched_at 기준)
```

#### 완료 기준
- [ ] 유효 토큰 → 통합 리포트 정상 렌더링
- [ ] 무효/위조 토큰 → 404
- [ ] 5개 섹션 + 면책 + 기준일 모두 표시
- [ ] accessed_at 기록

### 4-4. 결제 안내·환불 문구 (P0) ← 스레드 팁 4 반영

#### 목표
결제 전후로 "무엇이 열리고, 환불·복원은 어떻게 되는지"를 명확히 안내해 출시 후 문의·분쟁을 줄인다.

#### 결제 페이지 명시 항목
```
- 결제 금액: 9,900원 (1회성, 주소 1개 분석)
- 결제 후 제공: 입력하신 주소의 통합 분석 리포트 (즉시 열람)
- 열람 기간: 발급 후 [N]일간 동일 링크로 재접속 가능
- 환불 정책: 디지털 콘텐츠 특성상 리포트 열람 후 환불 불가.
             열람 전(미접속) 상태에서 [N]시간 이내 요청 시 환불 가능.
- 문의: [문의 채널]
```

> 환불 정책 문구는 법적 효력 검토 필요. 전자상거래법상 디지털 콘텐츠 환불 규정 확인 권장 (본 PRD는 법률 자문이 아님).

#### 완료 기준
- [ ] 결제 페이지에 금액·제공물·환불·문의 4종 명시
- [ ] 결제 완료 페이지에 "무엇이 열렸는지 + 재접속 방법" 안내

### 4-5. 지도 기반 객단가 시각화 (P1)

#### 목표
첨부 레퍼런스(이안 분석기, AirROI Atlas)처럼 카카오맵 위에 핀과 수익 라벨을 띄우고, 클릭 시 상세를 보여준다. **무료 미끼**로 작동(지도 열람은 무료, 정밀 리포트는 유료).

#### 데이터 출처별 노출 정책 (변호사 상담 후 확정, 2026-05)
| 출처 | 개별 노출 | 방식 |
|------|-----------|------|
| 공공데이터 외도민 | 가능 | 핀에 상호·주소 표시 (연락처·대표자 실명 제외) |
| AirROI 수익·예약률·객단가 | **금지** | 통계 가공값만. "이 동네 평균 N만원" 형태로만 |
| 에어비앤비 숙소 사진 | **금지** | 변호사 "이의제기 분명히 들어온다" 권고 |

> **근거:** 변호사 상담(2026-05) 결과, 개별 숙소 수익·예약률은 영업비밀 해당 가능성 있음. Jason(AirROI) 허가는 라이선스 문제 해결이고, 한국 영업비밀법은 별개. 통계 가공 원칙으로 재확정.

#### 지도 구조
```
KakaoMap
  ├── MarkerClusterer (외도민 공공데이터 핀)
  │     줌 아웃: 클러스터 숫자 ("55")
  │     줌 인:   개별 핀 + CustomOverlay
  └── 핀 클릭 → 정보창
        ├─ (공공) 상호 / 주소
        └─ (AirROI) 주변 동네 평균 수익·ADR·점유율
```

#### 완료 기준
- [ ] 카카오맵에 외도민 핀 + 줌 레벨별 클러스터링
- [ ] 핀 클릭 시 정보창 (동네 평균 우선)
- [ ] 지도 화면은 무료 접근, 정밀 리포트 CTA 노출

### 4-7. 입력값 검증 (P1) ← 스레드 팁 5·6 반영

#### 목표
주소·금액 입력에 이상값/스팸이 들어와도 안전하게 처리하고, 에러 메시지로 내부 구조를 노출하지 않는다.

#### 검증 항목 (서버측 필수)
| 입력 | 검증 |
|------|------|
| 주소 | 빈값/과도한 길이/스크립트 차단. 카카오 주소검색 결과만 허용 |
| 월세·초기비용 등 금액 | 숫자만, 음수·비현실적 큰 값 차단 |
| report_token | UUID 형식 검증 |

#### 에러 메시지 정책 (스레드 팁 6)
- 사용자에게는 일반화된 메시지만 ("일시적인 오류가 발생했습니다")
- DB 구조·서버 경로·스택트레이스는 **절대 클라이언트로 내보내지 않음** → Sentry에만 기록

#### 완료 기준
- [ ] 비정상 입력 차단 (주소·금액)
- [ ] 에러 응답에 내부 정보 미포함
- [ ] 상세 에러는 Sentry에만 기록

---

### 4-8. 안내 문구 외부화 (P2) ← 스레드 팁 7 반영

#### 목표
면책문구·문의 링크·공지를 코드에 하드코딩하지 않고 상수 파일(또는 DB)로 분리해, 문구 수정 시 재배포 부담을 줄인다.

#### 구현
- `src/constants/messages.ts`에 면책문구·문의링크·환불정책 텍스트 집약
- 컴포넌트는 이 상수를 import해서 사용

#### 완료 기준
- [ ] 면책·문의·환불 문구가 단일 상수 파일에 집약
- [ ] 문구 변경이 컴포넌트 코드 수정 없이 가능

## 5. 데이터 모델 (신규 테이블)

### `airroi_cache`
```sql
create table airroi_cache (
  id          uuid primary key default gen_random_uuid(),
  lat         numeric(10,6) not null,
  lng         numeric(10,6) not null,
  radius_m    int not null default 1000,
  data        jsonb not null,             -- AirROI 응답 원본
  fetched_at  timestamptz not null default now(),
  expires_at  timestamptz not null        -- fetched_at + 90일
);
create index on airroi_cache (lat, lng, radius_m);
```

### `report_purchases`
```sql
create table report_purchases (
  id              uuid primary key default gen_random_uuid(),
  polar_order_id  text unique not null,    -- 웹훅 중복 발급 방지
  address         text not null,
  lat             numeric(10,6),
  lng             numeric(10,6),
  report_token    text unique not null,    -- URL 접근 토큰
  purchased_at    timestamptz not null default now(),
  accessed_at     timestamptz              -- 첫 열람 시각
);
create index on report_purchases (report_token);
```

### `airroi_usage` (비용 모니터링)
```sql
create table airroi_usage (
  id          uuid primary key default gen_random_uuid(),
  called_at   timestamptz not null default now(),
  endpoint    text not null,
  cache_hit   boolean not null default false
);
create index on airroi_usage (called_at);
```

> RLS: 세 테이블 모두 클라이언트 직접 접근 차단. 서버(service role)에서만 read/write.

## 9. 개발 순서 (Claude Code 진입 순서)

바이브코딩 치트키 — One-at-a-time + Think-Then-Do 준수. 각 Step은 별도 프롬프트로.

```
Step 1. DB 스키마
  → airroi_cache, report_purchases, airroi_usage 생성 (RLS 포함)

Step 2. AirROI 데이터 레이어
  → src/lib/data/airbnbData.ts (캐시 + 비용 캡)
  → 입력값 검증 유틸 포함

Step 3. 통합 리포트 페이지 (먼저 무료 데이터로 골격)
  → app/report/[token]/page.tsx
  → 토큰 검증 → 5개 섹션 렌더링 (AirROI는 Step 2 함수 사용)

Step 4. Polar 결제 연동
  → @polar-sh/nextjs 설치
  → app/api/checkout/route.ts
  → app/api/webhooks/polar/route.ts (서명 검증 + 토큰 발급)
  → 결제 전 확인창 + 안내·환불 문구

Step 5. 지도 시각화
  → app/map/page.tsx (카카오맵 + 클러스터링 + 핀 클릭)

Step 6. 모니터링·알림
  → Sentry 연동
  → AirROI 실패/급증 알림, 웹훅 실패 알림

Step 7. 안내 문구 외부화
  → src/constants/messages.ts

Step 8. 배포 및 검증
  → Vercel 배포 → 실제 결제 1건 테스트
```

> 결제·중요 동작(Step 4)은 작업 전 Think 단계로 위험 요소 먼저 점검 (스레드 팁 7 정신).
