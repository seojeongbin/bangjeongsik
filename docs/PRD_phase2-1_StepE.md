# PRD — 주소 리포트 강화: 베드룸별 수익 데이터 (Phase 2-1 Step E)

> 배경: PRD_phase2-1.md(v2) 7번 "이번 개정 제외(별도 PRD)" 항목 중
> "인원·방·화장실별 객단가 필터"를 다룸.
> 선행: Step A~D-2 완료 (동 무료 미끼 → 주소 결제 깔때기 + 지도 UX 완성)
> 상태: 사전조사 완료, 작성 v1

## 1. 배경 — 현재의 한계

현재 모든 주소 리포트는 AirROI `/calculator/estimate`를
`bedrooms=2&baths=1&guests=4` 고정값 하나로만 호출함
(`src/lib/data/airbnbData.ts`). 사용자가 실제로 운영하려는 방 구조가
원룸이든 쓰리룸이든 동일한 추정치를 받음 — 실제 차이가 있는데도
반영이 안 됨.

## 2. 사전조사 결과 (2026-06-27 검증, 서교동 좌표 기준 6개 조합 실제 호출)

| 지표 | 변별력 | 근거 |
|---|---|---|
| ADR (1박 객단가) | ✅ 있음 | 1BR(121,446원) → 4BR(309,093원), 범위 187,648원 |
| 월수익(revenue) | ✅ 있음 | ADR과 동일 패턴으로 분화 |
| Occupancy(예약률) | ❌ 없음 | 54.2%~55.2%, 범위 0.9pp — 스펙과 거의 무관 |

**결론**: ADR/수익은 bedrooms별로 다르게 보여줄 가치가 있음. Occupancy는
스펙별로 나누지 않고 "이 동네 평균"으로 단일 고정 표시 — 스펙별로 나누면
"방 늘리면 예약률도 오른다"는 오해를 유발할 위험이 있어 의도적으로 분리.

이 패턴은 Step C(동 단위 occupancy 광역 스무딩 확인)와 다른 메커니즘.
주소 좌표는 고정이고 스펙(bedrooms)만 바뀌는 조건이라, ADR은 스펙에
실제로 반응하지만 occupancy는 지역 수요 지표라 스펙과 독립적인 것으로
해석됨 (버그 아님, AirROI 모델의 정상 동작으로 판단).

## 3. 기능 정의

### 3.1 무엇을 보여주나
주소 리포트 페이지에 신규 섹션 "베드룸별 예상 수익" 추가:
- 사용자가 선택한 bedrooms 1개에 대해 ADR, 월수익(P25~P75 포함) 표시
  (최초 1회 호출)
- 다른 bedrooms 비교는 사용자가 명시적으로 트리거할 때만 추가 조회
  (3.2 참고)
- Occupancy는 기존처럼 "이 동네 평균 예약률" 단일값으로 별도 표시
  (스펙별로 나누지 않음 — 이미 보유한 동네 단위 값 재사용, 추가 호출
  불필요)

### 3.2 사용자 입력 — 선택 후 단일 조회 (동시조회 아님)

**[2026-06-27 수정] 최초 설계는 bedrooms 1/2/3/4+ 4종류를 리포트 로드 시
동시 조회였으나, 캐시 구조를 검토한 결과 캐싱은 "반복 호출"만 막아줄 뿐
"최초 호출의 동시성"은 줄여주지 못함이 확인됨 — 즉 캐시 키를 고쳐도
리포트 1건당 최초 호출 수는 그대로 4배. 이에 동시조회 방식을 철회하고
아래로 변경:**

- bedrooms: 사용자가 1/2/3/4+ 중 하나를 **선택**(기본값은 운영 형태
  추정값 또는 2bedrooms 등 합리적 기본값 — 설계 단계에서 결정). 선택한
  1개에 대해서만 즉시 호출 — 최초 진입 시 호출 1회로 기존과 동일
- 다른 bedrooms 결과가 궁금하면 "다른 베드룸도 비교해보기" 등 명시적
  트리거(버튼/탭 클릭)로 그 시점에만 추가 호출. 사용자가 안 누르면
  추가 비용 없음
- baths, guests: 사용자가 직접 선택 가능. 단, 우리 쪽 자체 검증으로
  제한:
  - baths: 1 ~ 3 (0.5 단위, 즉 1, 1.5, 2, 2.5, 3)
  - guests: 선택한 bedrooms 값 이상만 허용 (bedrooms=1 → guests 최소 1,
    bedrooms=4 → guests 최소 4)
  - AirROI 공식 문서에 명시된 파라미터 유효범위를 못 찾아 우리가 상식
    범위로 선제 제한. API가 더 넓은 범위를 허용해도 무방, 혹시 더 좁은
    제약이 있다면 기존 에러 핸들링(DATA_UNAVAILABLE)으로 흡수
  - baths/guests를 바꿀 때도 매번 즉시 재호출하지 않고, "적용" 버튼
    또는 디바운스 후 1회 호출 — 사용자가 슬라이더/드롭다운을 여러 번
    조작해도 호출은 최종값 1번만

### 3.3 기존 시뮬레이터와의 관계
- 이 섹션은 **정보 제공 패널**, 기존 `ReportSimulator`(사용자 직접
  입력)는 **독립된 도구**로 그대로 유지
- 자동 연동(필터 선택값이 시뮬레이터 입력칸에 자동 반영) 안 함 —
  사용자가 패널에서 참고값을 보고 시뮬레이터에 본인이 직접 입력하는
  흐름

## 4. 비용/캐싱

**캐시가 해결하는 것과 해결하지 못하는 것을 구분해야 함**: 캐싱은
"같은 좌표+같은 bedrooms 조합을 누군가 90일 내 또 조회하는 경우"의
중복 호출만 막아줌. "한 사람이 리포트 1건을 열 때 몇 번 호출이 나가는가"
(최초 호출의 동시성)는 캐시와 무관하게 설계(3.2)에서 결정됨. 이 둘을
혼동해 "캐시 키만 고치면 동시조회도 1회로 줄어든다"고 가정하면 안 됨
— 검토 결과 그렇지 않음이 확인되어 3.2를 "선택 후 단일조회"로 변경함.

**호출 비용 비교 (변경된 설계 기준)**:

| 시나리오 | 기존(고정 1종류) | 변경 후(선택 후 단일조회) |
|---|---|---|
| 신규 좌표 첫 결제 1건, bedrooms 1개만 봄 | 1회 | 1회 (동일) |
| 같은 건, 다른 bedrooms도 비교해봄 | N/A | 1 + 비교 클릭 횟수 |
| 같은 좌표 90일 내 재방문(같은 bedrooms) | 0회(캐시 히트) | 0회(캐시 히트, 키 수정 후) |
| 같은 좌표 90일 내 재방문(다른 bedrooms) | N/A | 1회(그 bedrooms는 캐시 미스) |

**필수 변경 — 캐시 키 확장**: `airroi_cache` 테이블의 캐시 키가
`(lat, lng, radius_m)`만 사용 — `bedrooms`가 키에 없음. 사용자가
bedrooms를 선택하게 되는 이상, 같은 좌표라도 bedrooms가 다르면 다른
결과여야 하므로 캐시 키에 `bedrooms`(및 `baths`, `guests` 또는 이를
정규화한 조합 키) 추가가 필수. 이건 동시조회 여부와 무관하게 필요한
변경 — 단일조회로 가도 bedrooms가 가변값이 되는 순간 캐시 키 확장은
그대로 필요함.

**비용 결론**: 최초 진입 호출은 기존과 동일(1회) 수준으로 유지됨.
추가 비용은 사용자가 "다른 베드룸 비교"를 실제로 클릭할 때만, 그
클릭 횟수만큼만 발생 — 안 누르는 사용자에게는 추가 비용 없음.

## 5. 표시 위치 및 UI 방향 (상세 디자인은 구현 단계에서)

- 기존 SectionCard 패턴("동네 수익 데이터 — AirROI 통계") 톤 유지
- bedrooms 선택은 탭 또는 버튼 그룹(1/2/3/4+) — 기본 선택값 1개만
  먼저 호출, 다른 탭 클릭 시 그 시점에 추가 호출(로딩 상태 표시 필요)
- baths/guests 선택 UI는 select 또는 +/- 스테퍼, AirROI 유효범위
  제약을 selectbox 옵션 자체에서 제한(잘못된 값 선택 불가능하게).
  변경 즉시 재호출하지 않고 "적용" 버튼 또는 디바운스로 호출 1회 보장

## 6. 제외 항목 (이번 PRD 범위 아님)

- AirROI 매물개수(comparable count) 지표 — `/calculator/estimate`
  응답에 해당 필드 없음 확인. `/markets/summary` 등 별도 엔드포인트
  조사 필요, CLAUDE.md "[조사 필요]" 항목으로 별도 관리
- 인테리어 프리미엄 점수, 카카오 로그인 등 — PRD v2 7번에 명시된
  별도 분리 항목

## 7. 개발 순서 (제안)

- Step E-1: `airroi_cache` 캐시 키 확장(bedrooms 포함) — 스키마/로직
  변경, 기존 캐시 데이터 마이그레이션 또는 TTL 만료 후 자연 교체 결정
- Step E-2: `getAirbnbData` 함수에 bedrooms/baths/guests 파라미터화
  (선택된 단일 조합 기준 호출)
- Step E-3: 리포트 페이지에 "베드룸별 예상 수익" 섹션 UI 추가
  (bedrooms 선택 UI + 최초 1회 호출)
- Step E-4: baths/guests 사용자 선택 UI + 자체 유효범위 검증 +
  변경 시 디바운스/적용버튼으로 호출 1회 보장
- Step E-5: "다른 베드룸 비교해보기" 트리거 UI — 클릭 시에만 추가
  bedrooms 호출
